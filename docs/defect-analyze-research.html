1. 导读：我们要解决什么

Torin 当前的 analyze-defect agent（packages/agent/src/agents/analyze-defect/）在 20 个 turn 内同时承担六件事：

  1. 理解 defect 的意图（用户在抱怨什么）
  2. 探索仓库结构（什么技术栈、什么目录约定）
  3. 定位根因（具体是哪个函数 / 哪行代码）
  4. 设计修复方案（怎么改）
  5. 给出风险与范围（riskClass + scopeDeclaration）
  6. 产出验证步骤（HITL reviewer 用什么 checklist）

问题场景：tapd 中文 defect description 通常只有 2–5 句话，缺 expected/actual、缺复现步骤、缺涉及组件。在大型 monorepo 上，agent 没有起点，盲探目录，turn budget 耗光后强行 submit_result。结果：

  • rootCause 含糊（"可能与 XX 模块有关"）
  • scopeDeclaration 偏宽（把整个目录列上）或偏错（漏掉真正要改的文件）
  • 下游 implement / filter / critic 拿到劣质输入，错误被放大

本次调研的核心问题：

问：在 LLM-agent 做 SWE 任务这条路线上，2025–2026 年的论文与生产系统对"大仓上从短描述到准确 fix"这件事有什么经过验证的招？哪些是噪音、哪些是真信号？

我并行派了三个研究 agent，分别覆盖：(a) SOTA 大仓 fault localization、(b) bug report → intent extraction、(c) 多候选 fix 探索与选择。本文档把三份报告的发现拆解后，按"逐条论据 + 引用论文 + 对我们的启示"重新组织。

阅读建议 直接看 TL;DR 再翻"证据"各节深入。每节末尾有"对我们的启示"小段，把抽象论文映射到 Torin 当前代码。

2. TL;DR — 五条结论

  1. 采纳结构化 intent 前置有效，但必须产出可执行约束（grep queries / reproducer hypothesis / 成功判据），而不是文字总结。证据：SpecRover、AEGIS、CodeR、Google 的 decomposition 论文。
  2. 采纳仓库导航图（repo map）作为分析 agent 的缓存前缀。最佳形态：Tree-sitter + PageRank（Aider 风格），1–2k tokens，prompt cache 命中。证据：LocAgent、Aider、Trae CKG、LingmaAgent。
  3. 放弃之前提的"三个独立 sandbox agent 串联"方案，合并为 1 个 text-only intent agent + 1 个升级版 analyze agent。理由：Cognition 的 "Don't Build Multi-Agents" 警告 + multi-agent 总结式 handoff 的 telephone game。
  4. 放弃"枚举 N 个 diff 让 LLM 挑"。LLM 在 diff 层会 mode collapse（Verbalized Sampling 量化证明），LLM-as-judge 在相似候选间不稳定（AXIOM）。要选择就用 sandbox 跑回归 + 复现测试。
  5. 观察中文 tapd defect 是真空地带——没有论文专门覆盖。可移植实践：双语 key term 保留、项目术语表注入、显式 unknowns 字段防幻觉。

3. 背景知识（如果你已经熟悉可跳过）

3.1 SWE-bench / SWE-bench Verified

由 Princeton + Anthropic 合作整理的、来自 12 个真实 Python 项目的 2 294 个 GitHub Issue → Pull Request 配对。SWE-agent 给定 issue 文本，要求 agent 在仓库上自己定位 + 改代码 + 让真实测试通过。Verified 是 OpenAI 与 Princeton 人工筛过的 500 题子集（去掉了原数据集中题目模糊或测试错误的样本），现在事实上是 SOTA 比较的标准基准。Verified leaderboard

关键之处：题目原文是英文 GitHub Issue（通常已经包含 expected/actual + 复现脚本），仓库是中等规模 Python 库。tapd 中文短 defect + 大型多语种 monorepo 比 SWE-bench 难一个数量级。

3.2 Fault localization（FL）三种主要范式

范式代表对 LLM agent 的契合度
SBFL（Spectrum-Based）Ochiai / Tarantula，用测试通过/失败的覆盖差异打分需要先有失败测试，对短描述 defect 不直接适用
IRBFL（IR-Based）BLUiR、BugLocator，把 bug 文本和源码做 IR 匹配（BM25 / embedding）BM25 alone 在 SWE-bench Verified 上 floor 在 1.96%
Agent-basedSWE-agent、Agentless、AutoCodeRover、LocAgent当前 SOTA 全在这条路线上

3.3 几个关键术语

Pass@1 / Pass@N
SWE-bench 评估指标：单次提交解决 / N 次提交里至少一次解决的比例。
Best-of-N
同一题跑 N 次，挑一个最好的提交。提升来自 N 次结果的"上限"被某种 selector 取到。
Telephone game（电话游戏）
多 agent 之间通过总结字符串传递信息，每一次"压缩 → 解压"都丢细节，类似真实电话游戏。Cognition 用它形容多 agent 系统的失败模式。
Prompt cache（Anthropic 5 分钟 cache）
固定前缀走 cache 后输入 token 单价降 90%。设计上凡是"每次都重发的大段内容"（比如 repo map）应当尽量做成固定前缀。
Mode collapse
后训练 LLM 在采样时倾向给出几乎相同的答案，即使温度调高、要求多样化。Zhang et al. 2025 Verbalized Sampling 给了量化测量。

4. 证据 1 — Intent 前置真的有用吗？

核心问题：在让 agent 跑代码之前，单独跑一段 "理解 bug 文本" 的步骤，有没有提升下游 fix 成功率？

4.1 SpecRover（ICSE 2025）— 最直接的对照实验

SpecRover 是 AutoCodeRover 的升级版，作者团队相同（NUS Abhik 教授组）。流水线五个 agent：

Reproducer  →  Context Retrieval  →  Patching  →  Reviewer  →  Selection

关键差别：Context Retrieval 阶段除了输出 "改哪一行"，还输出每个候选位置的 natural-language specification（即"这段代码本来应该满足什么不变量"）。下游 Patching agent 收到的不是单纯的位置列表 [loc_1, loc_2, ...]，而是 [(loc_1, spec_1), (loc_2, spec_2), ...]。Reviewer 在多个候选 patch 之间选择时，用 原 issue 文本 + 推断的 spec 作为对照，覆盖"测试通过但 spec 错"的 patch。

对比数字：在 SWE-Bench Verified 上 +50%（pass@1 从 ~30% → ~46.2%），单 issue 成本反而降到 $0.65（vs AutoCodeRover ~$0.50–0.90 但成功率低）。arxiv:2408.02232 / ICSE'25 PDF

"Issues often contain richer information than tests alone... A spec derived from the issue can override misleading test signals."
— SpecRover §3.2

对我们的启示

SpecRover 把"intent"做成下游每个步骤都能引用的 artifact，而不是 "intent agent 跑完就丢了"。这是我们 R2 方案里 intent 前置的核心理由：intent 不是中间消费品，是固定贯穿整条 pipeline 的约束。

4.2 AEGIS（FSE 2025）— Ablation 直接证明"约束作用"

AEGIS（"Augmenting bug reports with Expected behaviors using GenAI for Improved fault localization Statistics"）做了一个非常干净的对照：

条件MAP@10 提升
原始 bug report（baseline）—
+ LLM 生成的 summary 作为额外文本注入+4.3 pp
+ summary 作为下游 IR ranker 的显式 query 约束+23.7 pp

解读：差异不在"有没有 summary"，而在 summary 怎么被使用。把它当文本拼进去，得 4.3 pp；把它解析出关键词 → 作为 IR ranker 的硬约束，得 23.7 pp。arxiv html

对我们的启示

这是我们坚持 intent agent 必须输出searchHypotheses: [{ hypothesis, queries }]（具体可执行的 grep / glob 查询）而不是 prose 的核心理由。如果只产 summary，价值大概只有 4 pp 级别。

4.3 CodeR（Lin et al. 2024）— Pre-planning vs Reactive

CodeR 是多 agent 系统，包含 Manager / Reproducer / Fault Localizer / Editor / Verifier。和其他多 agent 系统的差异在于：有一个固定的 task graph 提前规划好，agent 按图执行而不是自主决策"下一步做什么"。

核心 ablation：固定 task graph vs 自主决策 → +11.3 pp（在 SWE-bench Lite 上 19% → 30%+）。arxiv:2406.01304

"Reactive agents waste turns deliberating control flow that's already determined by the task structure."
— CodeR §4

对我们的启示

Torin 已经用 Temporal workflow 做了 pre-planning（每个 stage 是固定的），这条路线已经走对了。继续在 stage 内部细化 pre-planning（intent → repo map → analyze）符合 CodeR 的发现。

4.4 Google "Small Models, Big Results" — Intent Extraction by Decomposition

Google Research 2025 的一篇博客，做的是 NLU 领域的 intent extraction（不是 SWE，但 idea 通用）。核心发现：与其用一个大模型一步到位提取 intent，不如把任务拆成多个小步骤、每步用小模型——成本降一个数量级、准确率不降甚至提升。research.google

对我们的启示

这是我们让 intent agent 用更便宜模型（Haiku 级别）的论据。Intent 阶段是文本→结构化，complexity 不需要 Opus 4.7。当前 Torin runAgent 只读全局 AGENT_MODEL，加 per-agent override 是低成本改造。

4.5 IQLoc — 结构化 keyword 抽取的"硬"实验

较老的工作（2022），但 ablation 干净：从 bug report 抽取关键词作为 IR query 后 MAP +60%。arxiv:2204.09655

4.6 综合：Intent 前置该做成什么样？

这些证据综合起来告诉我们的"应当"：
  1. Intent agent 输出可执行约束：search queries（直接 grep）+ reproducer hypothesis（可写成 stub）+ 成功判据（用户认为修好的判据）+ unknowns（不知道什么）。不要是 prose summary。
  2. Intent 是下游每一步都能引用的固定 artifact，注入到 analyze、reproduce、implement、critic 的 prompt 里都该有它（SpecRover 模式）。
  3. Intent 步骤本身是一次轻量 LLM 调用，不是 agent loop。无需 sandbox 工具，~6 turns。可以用便宜模型。
  4. 必须有 unknowns 显式通道，避免 LLM 在缺字段时幻觉补全（见 反向证据 中 CTQRS 部分）。

5. 证据 2 — Repo Map / 仓库导航图

核心问题：大仓上 agent 没有起点。怎么给它一张"地图"？

5.1 LocAgent（ACL 2025）— 把仓库变成图，让 agent 走图

LocAgent 把仓库构造成一个 directed heterogeneous graph：

  • 节点：目录、文件、类、函数
  • 边：contain（结构）、import（依赖）、invoke（调用）、inherit（继承）

暴露给 agent 三个工具：

  1. SearchEntity(query)：BM25 检索，返回三层粒度（fold = 只名字 / preview = 签名+docstring / full = 完整代码）
  2. TraverseGraph(start, edge_type, hops)：类型感知的 BFS（"找所有调用 foo 的函数，2 跳内"）
  3. RetrieveEntity(id)：获取实体详情

结果（SWE-Bench Lite）：

指标LocAgent (Qwen-2.5-32B 微调)对比
file-Acc@592.7%BM25-only ≈ 60%
function-Acc@1077%SWE-agent (GPT-4) ~ 50%
成本$0.05 / issueSWE-agent ~$0.30

更重要的是 ablation：单独砍掉 invoke / import / inherit 边，function-level accuracy 大幅崩塌。结构边才是多 hop 多回报的关键。arxiv:2503.09089 / github

对我们的启示

完整的 LocAgent 实现需要 tree-sitter + 自建图存储 + 自建查询工具，工程量大。但它的 核心 insight 适用：仅仅把仓库平铺给 agent grep，不如先建结构。我们短期 R2 用更轻量的 Aider repo map（见 5.2），中长期可考虑 LocAgent 形态。

5.2 Aider Repo Map — 实用、轻量、已开源

Aider 的 repo map（2023 年首发，至今仍在迭代）做法：

  1. 用 Tree-sitter 解析每个源文件，提取 definitions（class、function、method）和 references（调用关系）。Aider 内置 tags.scm 覆盖 ~20 种语言。
  2. 每个文件作为图节点，definitions/references 构成边。
  3. 跑 PageRank：被引用越多的符号权重越高（"中心节点"）。
  4. 按权重选 top-N 符号，渲染成简洁文本： 
    src/cart.ts:
│class ShoppingCart:
│    def add_item(item: Item) -> None
│    def remove_item(item_id: str) -> None
⋮
src/checkout.ts:
│def process_payment(cart: ShoppingCart) -> PaymentResult
  5. 默认 1 024 tokens 预算，可调。

关键设计：map 是 chat 的固定前缀，每次对话开头同样的 map，吃 prompt cache。这是 Aider 在长会话里保持便宜的核心机制之一。

对我们的启示

Torin 已有 analyze-repository agent 跑过一次产出仓库结构性 AnalysisResult。短期可以：

  • R2 第一步：直接拿 AnalysisResult 里 techStack / 关键目录 / patterns 作为 navigation hint（~500 tokens）。
  • R2 第二步（可选）：让 analyze-repository 额外跑 tree-sitter + PageRank 产出 Aider 风格的 repo map，作为 AnalysisResult 的新字段。

无需向量库、无需图数据库，纯文本，~2k tokens，走 prompt cache。

5.3 Bytedance Trae Agent（2025）— 生产系统验证

Trae（即 MarsCode 的延续）暴露给 agent 的工具：

  • str_replace_editor + bash（基础读写）
  • ckg_tools：Code Knowledge Graph，包含 search_class(name) / search_function(name)
  • sequential_thinking_tool（强制 step-by-step 写思考）

SWE-Bench Verified 单次成功率 60.6–62.8%，5 并发 + 三阶段选择（ensemble vote → regression filter → o1 picks best）→ 68.2%，一度榜首。Trae 博客

对我们的启示

CKG（Code Knowledge Graph）就是 LocAgent 那套思想的工程化版本。两点确认：

  • 大厂榜首系统都建了图，没有人靠纯 grep
  • 没有人在 SWE-bench 顶部用 LSP。LSP 理论最优，但生产工程负担太大（per-language server bootstrap + cold-start indexing + Docker 内可靠性），大家都自建 tree-sitter 图

5.4 Alibaba LingmaAgent（FSE'25 Industry Track）— 唯一真实生产数据

LingmaAgent 是阿里云内部部署的版本，论文给出真实业务数据（Java / TS / JS）：

  • 16.9% 全自动
  • 43.3% 在 < 5 次人工介入下完成
  • function-level accuracy 49.3% / file-level 67.7%

架构：top-down AST → KG 构建，然后 UCT-based MCTS 探索：BM25 在扩展阶段做优先级、LLM reward agent（CoT，0-10 打分，≥6 留下）在 simulation 阶段、调用/被调用邻域增长在 backprop。arxiv:2406.01422

对我们的启示

非 Python 大型多语言仓上唯一公开生产数据。Function-level 49% 给我们一个心理基线——我们不应期望 SWE-bench Python 上 70% 的数字直接迁移到 tapd 大仓。

5.5 反例：Claude Code 为什么不建索引

Claude Code 的设计 retrospective 直接拒绝了向量库：

"Early versions used RAG with a local vector database, but we found pretty quickly that agentic search generally works better."
— vadim.blog · Claude Code, no indexing

Claude Code 的策略：三层 Glob → Grep → Read，需要时启动子 agent（Haiku，隔离 context）做"深探"。靠 92% prompt cache 命中率维持经济性，没有任何持久化索引。

对我们的启示

不要做向量库。两个理由：

  1. Multi-SWE 上 BM25-alone floor 在 1.96%，embedding 帮忙也只是 marginal
  2. per-issue snapshot 语义下，按 commit 临时 tree-sitter 比维护一个会过期的索引更省事更新鲜

5.6 综合：Repo Map 该做成什么样？

方案选型：
  • 短期（本次重构）：复用现有 AnalysisResult，挑出 techStack / keyDirectories / patterns 大概 500–1000 tokens 作为 navigation hint 注入 analyze prompt。
  • 中期：在 analyze-repository agent 里加 tree-sitter + PageRank 生成 Aider 风格 repo map，写到 AnalysisResult 一个新字段，~2 k tokens。
  • 长期（半年级别）：考虑 LocAgent 式图存储 + 显式 graph traverse 工具。本次不做。
不做：向量库 / embedding 索引 / LSP server / 持久化索引数据库。

6. 证据 3 — Reproducer-first 的位置之争

核心问题：跑测试复现 bug 这件事，该放在 pipeline 哪儿？

6.1 MarsCode / Trae — 把 Reproducer 放最前

MarsCode 是 SWE-Bench Lite 当时的 #1（34%），架构：

Reproducer  →  Repo Browser  →  Code Editor  →  Test Analyst

Reproducer 是 pipeline 的第一站，先尝试用 issue 文本生成一个可执行的 reproducer test。能跑通的话，stack trace 直接成为 localization 的强信号。arxiv html

6.2 SpecRover — Reproducer 在 Context Retrieval 之前

同上结构：Reproducer → Context → Patch → Review → Select。Reproducer 输出的 trace 是 Context Retrieval 的输入。arxiv:2408.02232

6.3 Agentless — Reproducer 用来选 patch，不用来定位

Agentless 不用 reproducer 做 localization；它用 reproducer 在最后阶段过滤 patch（"应该让 reproducer 通过的 patch 才留下"）。arxiv:2407.01489

6.4 反例：AutoCodeRover django-139645 case study

AutoCodeRover 论文里专门讲了一个失败案例：从 bug 描述抽出的 reproducer 涉及类 A，但真实 buggy class 是 B（A 只是触发路径上的一环）。Reproducer 把 patcher 引偏到 A，最终 patch 错误。arxiv pdf §6

对我们的启示

Reproducer-first 的好处明显（stack trace 是免费的精准信号），但前提是 defect 描述清晰到能写出 stub。tapd 短中文描述常常不够。两条路：

选项 A（保守） 保持当前顺序 analyze → HITL → reproduce。Intent agent 输出 reproducerHypothesis 字段（"如果要写 stub，大概会长这样"），但实际 reproduce 留在原阶段。

风险低，改动小。
选项 B（激进） 在 analyze 内部尝试 lightweight reproduce（哪怕只 grep 错误消息看哪里抛），用收获的信息精化 localize。

有 AutoCodeRover django case 风险，把 reproducer 失误传播到 localize。

R2 方案取选项 A。Reproducer 顺序留到 Phase 后续再评估。

7. 证据 4 — 测试是唯一可靠的 patch 选择信号

核心问题：如果我们让 LLM 同时生成多个候选 patch，怎么选？

7.1 Agentless 的 ablation（决定性证据）

配置SWE-Lite 通过率成本
单次贪心生成23.33%$0.11
+ 42 候选采样 + AST-标准化 majority vote26.00%$0.34
+ 回归测试 + 复现测试过滤27.33%$0.34
upper bound（所有 42 候选中"有一个对"）41.0%—

解读两点：

  1. 采样 + 投票 alone 涨 2.7 pp；加测试过滤再涨 1.3 pp。两者都不是大跳。
  2. Upper bound 41% 说明 14 pp 的提升留在桌上了——正确的 patch 经常在候选集里，但选择器没找到。这才是 SOTA 系统里最大的"未填坑"。

arxiv:2407.01489 / github

7.2 LLM-as-Judge 在 patch 选择上的不稳定性

2025 年好几篇专门评估 LLM-judge 的论文都得到类似结论：

研究结论
HITL LLM-as-Judge for Patches (2025)共识 patch 集 precision 0.80 / recall 0.94 / κ=0.75；歧义集 precision 降到 0.65 / κ 0.57
AXIOM (2025)judge 在 paraphrasing / verbosity / 格式扰动下会翻转，排序不稳定
Agentic Repair at Scale (2025)生产部署里 judge 能做 0.86 precision 的"过滤明显坏的"，但不能可靠 rank 相似候选

对我们的启示

不要做：让 LLM 在多个候选 patch 之间排序、择优、做最终决策。这正好是 judge 最不可靠的场景。
可以做：让 LLM 做"明显坏的"过滤（precision 0.86，recall 0.94 这个工作模式）。剩下的"好但难分"，交给 sandbox 跑回归测试和复现测试。

7.3 Trae 的三阶段选择 — 工业 SOTA 套路

Trae 5 并发跑 → 68.2%（vs 单跑 62.8%）。三阶段选择：

  1. Ensemble vote：5 个 candidate diff 用 AST normalize 后投票
  2. Regression filter：跑仓库自带的测试，砍掉破坏现存测试的
  3. o1 picks best：剩下的让强 model 做最终一票

关键：LLM judge 是最后一道过滤，且作用在已经被前两道过滤过的小集合上。Trae 博客

7.4 综合：candidate 选择该怎么做？

本次重构不引入 Best-of-N implement（那是 Phase 2 的事）。但要为未来铺路：让 analyze 输出的 consideredStrategies 数组未来能成为 Best-of-N 的种子，每个 strategy 独立生成 patch、回归+复现测试过滤、最后让 LLM 在 ≤2 个候选里挑（这是 LLM judge 可靠的小区间）。

8. 证据 5 — 探索 vs 利用：REx 的 bandit 视角

核心问题：有 N 个候选时，是"采样更多 fresh 候选"还是"refine 现有候选"？

8.1 REx（NeurIPS 2024）— 把 repair 形式化为 bandit

Tang et al. 把 LLM-driven program repair 建模为 arm-acquiring bandit：

  • 每个 candidate 是一个 arm；每个 arm 上有 Beta posterior 描述"它最终能通过测试"的概率
  • 每一步：要么 exploit（refine 当前最高 posterior 的 arm），要么 explore（refine 低 posterior 的 arm，或者拉一个全新 arm）
  • 用 Thompson Sampling 决定

结果：在 APPS-Competition、loop invariant、ARC 三个任务上，1.5–5× 更少 LLM 调用达到 Reflexion / 纯采样的准确率。NeurIPS PDF / github

8.2 REx 的两个反 intuition 启示

  1. 纯 "采样 N 个全新 candidate" 浪费 budget——很多 arms 的潜力很快就能判断（一两次执行测试），不需要每个都展开。
  2. 纯 "iterate 当前最好" 容易陷入 local optima——同一个错思路 reflection 多少次都改不出对答案。

对我们的启示

当前 Torin Phase 1.5 已经有 previousAttempts Reflexion 形态——iteration on best。REx 启示我们：当多个 attempt 都失败时，下一步应该是切换到另一个 strategy（探索），而不是继续 refine 这个 strategy（利用）。这恰好是我们 R2 把 consideredStrategies[] 做成 fallback 队列的理由。

具体流程：

implement attempt 1: 用 strategy[0]
  → 测试失败 → 添加 attemptMemo, refine strategy[0]
implement attempt 2: refine strategy[0]
  → 测试失败 → REx 提示：切换 arm
implement attempt 3: 改用 strategy[1]   ← 关键

9. 反向证据：哪些招要警惕

9.1 Cognition "Don't Build Multi-Agents"（2024-2025）— 反多 agent 工程

Devin 团队（Cognition）的 blog，根据自己产品迭代的失败经验总结。核心论点：

"Multi-agent systems compress full context into summaries when handing off between agents. Every summarization is lossy. We've found that a single agent with the full trace consistently outperforms a multi-agent decomposition with summary handoffs."
— cognition.ai/blog/dont-build-multi-agents

他们用 "telephone game"（电话游戏，每经一手信息就丢一些）形容多 agent 系统失败模式。具体反 pattern：

  1. agent A 跑完后只给 agent B 一段 JSON 总结
  2. agent B 看不到 A 的具体思考、看的代码、试过的 grep
  3. B 做决策时缺关键 context → 错误判断

对我们的启示

这条反例直接否决了我之前的 3-pass 方案（triage-defect-intent → localize-defect-root-cause → explore-fix-strategies，三个独立 sandbox agent 通过 zod schema 传递）。R2 改成：

  • Intent agent 独立保留（它只做格式转换，没 telephone game 风险——下游只需 intent 的输出，本来 intent 也没有可丢的"过程信息"）
  • Localize + strategy 枚举合并到同一个 sandbox agent run。Strategy 枚举作为 analyze 输出 schema 的字段，agent 在同一 conversation 里产出，无 handoff 损失。

9.2 Context Rot — 中段信息丢失

Anthropic 与多家学界做的 ablation 显示：长上下文里中段位置的信息，模型召回精度比首尾低 15–20 pp。

启示

Intent 注入到 analyze prompt 时，应当放在系统 prompt 末尾 + user prompt 开头这两个高召回区，不是塞在中间。

9.3 Vector DB / Embedding 索引仓库

三条独立证据反对：

  • Claude Code retrospective（5.5 节）
  • Multi-SWE-bench：BM25-alone floor 1.96%，embedding 也只是 marginal complement
  • per-commit snapshot 场景下，索引天然会过期

9.4 Mode Collapse — Verbalized Sampling 量化

Zhang et al. 2025 的 Verbalized Sampling 论文证明：post-trained LLM（包括 GPT-4o, Claude 3.5, Llama 3）在 T=0.8 时多次采样会高度相似，多样性 score 比"明确要求列 N 个不同方案 + 写出概率"低 1.6–2.1×。

启示

如果让 analyze agent 输出 "3 个候选 strategy"，prompt 写法很重要：

  • ❌ "请给出 3 个候选 strategy"（容易 mode collapse）
  • ✅ "请列出 3 个显著不同的 strategy，对每个标注你对它正确的信心（0-1），它们应代表不同的修复路径（例如：在 bug 处修 / 在上游守 / 在下游兜底）"

9.5 LLM-as-Judge 在排序相似候选时不稳定

AXIOM、CodeJudgeBench 等多个 benchmark 都证明：fine-tuned judge model 经常比 general CoT 弱；judge 对 verbosity、格式、paraphrasing 都敏感会翻转结果。

启示

已在 7.2 节展开，不重复。

9.6 CTQRS — LLM 在缺字段时幻觉补全

CTQRS 是 LLM 生成 bug report 质量的 benchmark；同时也指出 LLM 在 reformat / 结构化 bug report 时会编造"用户没说的"内容。如果让 LLM 提取 "expected behavior" 但原文压根没说，模型倾向于自己编一个"听起来对"的答案。

启示

Intent schema 必须有 unknowns: string[] 字段，并在 prompt 里奖励"如实标注 unknown"，明确禁止编造缺失字段。可以放 few-shot 示例：

defectDescription: "导出 Excel 金额不对"
DefectIntent:
  expectedBehavior: null
  actualBehavior: "导出的 Excel 中金额数据不正确"
  unknowns: [
    "金额是怎么错的（错位 / 单位错 / 精度丢失 / 全错为 0？）",
    "哪个导出入口（订单 / 报表 / 财务对账？）",
    "影响哪些用户角色"
  ]

10. 中文 tapd 场景的特定考虑

10.1 学术界真空

我们花了不少功夫查中文 bug 报告 / 内部研发系统的论文。基本结论：

  • Multi-SWE-bench（阿里 2025）覆盖多编程语言（Python / Java / TS / Go / Rust），不是多issue 语言。所有 issue 仍然是英文。
  • Tencent CodeBuddy 集成了 TAPD，但没有公开方法学。商业产品。
  • Bytedance Trae、Alibaba LingmaAgent 用的都是英文 SWE-Bench 来评估，部署在国内但没有针对中文 issue 的 ablation 公开。

结论：tapd 中文 defect → 修复 这条路上，没有先验论文可以照抄。我们的设计需要自己定义评估。

10.2 可借鉴的 cross-lingual 模式：CrosLocator

CrosLocator 是一个 cross-lingual bug localization 框架，做的是非英文 bug report 配英文代码仓库的检索。关键启示：不要全文翻译。中文 UI 文案（按钮标签、错误弹窗）是literal grep target——代码里可能有 i18n key 或硬编码字符串与之直接对应。翻译成英文会丢失这个匹配。

所以 intent agent 的处理规则应当是：

  1. 把整体描述翻译/归一化成结构化 summary
  2. 但 keyTerms 字段保留中文原文，作为后续 grep 查询的种子
  3. 同时保留可能的英文翻译（双语 keyTerms），用 OR 形式覆盖两边

10.3 项目术语表（domain glossary）注入

tapd defect 里经常出现产品/业务术语（"对账中心"、"PMI 计算"、"灰度配置"），对应到代码可能是某个具体模块名（reconciliation/、pmi-calculator/、canary/）。这种映射 LLM 凭空猜不准。

解决思路：在 analyze-repository 时让 agent 顺手产出一个 glossary: { 业务术语 → 代码概念 } 字段，缓存到 Project。Intent agent 在 prompt 里注入 glossary 后，能把"对账中心"映射到 src/reconciliation/。

这是本次重构可以做的低成本提升，但不在 R2 第一版强制要求——先把 intent 前置 + repo map 跑通看效果，glossary 是后续 phase。

10.4 unknowns 字段在中文场景的重要性

中文 tapd 短 defect 缺字段的比例比英文 GitHub Issue 高一个数量级。unknowns 字段 + 显式"禁止幻觉"prompt 不是 nice-to-have，是必须有。否则下游 analyze 会被一段编造的 "expected behavior" 引导到错误根因。

11. 修订方案 R2 逐段解读

把所有证据综合，得到这个 R2 形态：

analyze stage runs:
  ┌──────────────────────────────────────────────────────────────┐
  │ 1) intentBrief = triageDefectIntent(defectDescription)        │
  │    text-only, ~6 turns, 便宜模型                              │
  │    输出: { summary, expectedBehavior, actualBehavior,         │
  │            defectType, keyTerms (bilingual),                  │
  │            searchHypotheses, reproducerHypothesis, unknowns } │
  ├──────────────────────────────────────────────────────────────┤
  │ 2) repoMap = lookupRepoMap(projectId)                         │
  │    确定性活动（DB 查询），server 侧                            │
  │    取最新 ANALYZE_REPOSITORY 的 navigation 摘要 (1-2k tokens) │
  ├──────────────────────────────────────────────────────────────┤
  │ 3) analysis = analyzeDefect(sandbox, intentBrief, repoMap)    │
  │    sandbox read-only, ~25 turns                                │
  │    Intent + repoMap 作为 prompt 缓存前缀                       │
  │    输出 schema 扩展两个字段:                                   │
  │      candidateRootCauses[1-3]   ← 多个根因排序                │
  │      consideredStrategies[1-3]  ← Verbalized Sampling 风格    │
  │        each { approach, scopeFiles, tradeoffs, riskClass,     │
  │               recommendation, expectedFailureMode }            │
  ├──────────────────────────────────────────────────────────────┤
  │ 4) HITL gate (一次)                                            │
  └──────────────────────────────────────────────────────────────┘

11.1 为什么是 2 个 agent 而非 3 个

见 第 9.1 节 Cognition 反例。Intent → analyze 之间的 handoff 损失可控（intent 输出就是给下游用的结构化约束），但 analyze → strategy enumeration 之间共享强 context，独立 agent 会丢失"agent 看过的文件、试过的搜索、对哪些 hypothesis 已证实"等关键信息。合并到一个 agent run 里，让它在同一 conversation 末段产出 consideredStrategies。

11.2 为什么 intent 用便宜模型

见 第 4.4 节 Google 论文。Intent 是文本→结构化任务，complexity 不需要 Opus 4.7。Haiku 4.5 完全够用。当前 runAgent 只读全局 AGENT_MODEL，加 per-agent override 是低成本改造。

11.3 为什么 searchHypotheses 必须是 grep-able queries

见 第 4.2 节 AEGIS。Summary 作为文本拼接只值 4.3 pp，作为下游显式约束值 23.7 pp。Search hypotheses 是 analyze agent 第一步的 todo list（"先把这些 grep 跑一遍"），不是参考资料。

11.4 为什么 repo map 是确定性活动而非 agent

查 ANALYZE_REPOSITORY 历史输出是 DB 操作，本来就是确定性的。塞进 Temporal workflow 不需要 LLM 介入。这同时避免了 workflow 的 determinism boundary 问题（workflow 代码不能直接做 IO）——把查询放在 server 侧 ResolveDefectService.execute 里、查完结果作为 workflow input。

11.5 为什么不在 analyze 内嵌 reproducer

见 第 6.4 节 AutoCodeRover django case。Reproducer 失误会把 localize 引偏。当前 Torin 的 analyze → HITL → reproduce 顺序保留，intent agent 只输出 reproducerHypothesis（"如果要写 stub，大概会长这样"），实际 reproduce 留在 REPRODUCE 阶段。

11.6 为什么单一 HITL gate

三个 sub-stage 各自 gate 会让 reviewer 点击量×3，且与 CANONICAL_STAGE_ORDER 不兼容。Reviewer 在一屏看到 intent + candidateRootCauses + consideredStrategies 是更多信息，不是更少。Rejection 反馈下三段全重跑（intent 6 turns 很便宜）。

11.7 为什么 candidateRootCauses 和 consideredStrategies 都是 array

见 第 8 节 REx。Array 是为未来 implement 阶段的 fallback queue 准备的种子。本次 analyze 阶段只用 candidateRootCauses[0] 和 consideredStrategies[recommendedIndex] 驱动下游，但额外候选保留在 schema 里、写到 stage event，未来 implement 失败时 workflow 可以自动切换 strategy 而无需重跑 analyze。

11.8 与现有 Phase 路线图的关系

Phase已落地本次 R2未来
Phase 1（已落地）analyze schema 扩展（riskClass / scopeDeclaration / hasTestInfra / hasWebUI）+ reproduce agent 3 模式 + scope check + filter 多源 oracle——
Phase 1.5（已落地）Reflexion previousAttempts in implement + precondition guard——
R2（本次提案）—intent 前置 + analyze 升级（含 candidate/strategy 数组）+ repo map 注入—
Phase 2（已规划）—不动Best-of-N implement（用 R2 输出的 strategies 数组作种子）、critic agent、auto-approve trivial、repo memory

13. 术语表

Agent loop / Agent decision loop
LLM 自主决定下一步动作（read / grep / bash / submit）的循环。Torin 用 Claude Agent SDK 提供这个 loop，每个 stage 由一个 agent run 完成。
Activity（Temporal）
Workflow 里的 IO 操作单元。必须幂等。Torin 把 sandbox 操作、DB 操作、LLM 调用都放 activity，workflow 自己保持纯函数。
Best-of-N
同一题跑 N 次，挑一个最好。SOTA SWE-agent 系统几乎都用 Best-of-N 在 leaderboard 顶部刷分（典型 5-10 个 attempts）。
BM25
经典 IR 排序算法，基于词频和文档长度归一化。在代码检索上仍是强 baseline。
CKG（Code Knowledge Graph）
把仓库构造成图（class/function 节点 + 调用/继承 边）。Trae、LingmaAgent 都用类似结构。
HITL（Human-in-the-loop）
人工审批关键节点。Torin 的 analyze 和 final PR 都有 HITL gate。
Intent
本文档语境下，指对原始 defect description 的结构化理解：expected/actual behavior、defect type、key terms、search hypotheses、unknowns 等。
Mode collapse
LLM 多次采样时输出过度相似的现象，限制了 Best-of-N 的多样性收益。
MCTS（Monte Carlo Tree Search）
SWE-Search、LingmaAgent 等用来在 "下一步 action" 空间里做带剪枝的探索。
PageRank（在 repo map 语境下）
把代码符号当图节点、调用关系当边，跑 PageRank 得到符号"重要性"，用于挑选要放入 map 的 top-N。Aider 的核心做法。
Pass@1 / Pass@N
SWE-bench 指标。Pass@1 = 第一次提交就对的比例；Pass@N = N 次提交里至少一次对。
Prompt cache
Anthropic API 的功能：标记 prompt 前缀作为缓存项，5 分钟内重用降 90% 输入 token 成本。
REx
NeurIPS'24 论文，把 LLM repair 形式化为 arm-acquiring bandit，给出 explore vs exploit 的 principled 策略。
SBFL（Spectrum-Based Fault Localization）
用测试通过/失败覆盖差异打分定位 bug 行。需要先有失败测试。
Sandbox（Torin）
Docker 容器，clone 了目标仓库。agent 通过 MCP server 暴露的 bash / read_file / list_files 工具操作，不能访问 host 文件系统。
Scope declaration
analyze agent 输出的"允许 implement 改动的文件清单"，filter 阶段会做 mechanical 检查砍掉超出范围的 patch。
Stage（Torin workflow stage）
analyze / reproduce / implement / filter / critic / hitl / pr 七个 stage，每个 stage 一个 TaskEvent 记录。
Telephone game
Cognition 用来形容多 agent 总结式 handoff 信息丢失的比喻。
Tree-sitter
多语言增量解析器库。Aider、LocAgent 都用它生成代码符号图，无需运行项目自身的 build 系统。
Verbalized Sampling
2025 年技巧：让 LLM 明确"列出 N 个不同方案，每个给概率"，比朴素的高温采样多样性高 1.6-2.1×。
